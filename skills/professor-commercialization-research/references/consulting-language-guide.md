# Scholar-Shareable, MBA-Readable Consulting Language Guide

Use this guide for all visible 商业化分析 PPT titles, labels, bullets, chart captions, implications, and executive summaries. Use MBA-level comprehension as an internal readability benchmark while keeping the artifact suitable for direct sharing with the named scholar. Preserve technical rigor in evidence notes and the appendix; do not use academic phrasing as a proxy for rigor.

## 1. Reader and voice

Write so that an MBA-level generalist can assess customers, competition, unit economics, channels, execution, and risk without reading the papers. This describes the reading level, not visible audience copy.

Use a voice that is:

- **direct**: state the conclusion before background;
- **specific**: name the customer, workflow, deliverable, metric, and time period;
- **commercial**: translate value into revenue, cost, time, risk, quality, or experience;
- **bounded**: state where the conclusion applies and where it does not;
- **honest**: say what is proven, inferred, contradicted, or still unknown;
- **active**: identify who does what and what result follows.

Do not sound like:

- a paper abstract summarizing novelty and methods;
- a grant application promising broad social impact;
- a technical-transfer checklist listing every field;
- an investment pitch using unsupported superlatives;
- a consultant hiding weak evidence behind jargon.

### 1.1 Scholar-facing default

Unless the user names another recipient or requests an internal-only memo, assume the finished deck may be forwarded unchanged to the named scholar.

- Use the scholar's verified formal title or name in the cover and formal headings; reserve conversational honorifics such as `老师` for dialogue.
- Evaluate the evidence and readiness of a commercialization direction, never whether the scholar or the research is academically "worthy".
- Phrase gaps as missing customer, product, transfer, or execution evidence. Prefer `公开证据尚未支持客户付费` over `团队缺乏商业能力`.
- Use collaborative language such as `建议进一步确认`, `首轮验证需要回答`, and `希望与团队共同核实`.
- Keep reader personas and production logic invisible. Never print `面向 MBA 毕业生`, `普通商业决策者`, prompt text, workflow instructions, QA notes, model/tool names, or slide-production commentary.

For an externally shareable Chinese cover, prefer:

- `科研成果商业化分析`
- `[姓名][已核验职称]科研成果商业化机会初步分析`
- `研究全景、商业化方向与下一步验证建议`
- `基于公开资料整理｜供交流讨论`

Avoid cover language such as `先看清研究版图`, `判断哪些成果值得商业化`, `进入付费验证`, `最有价值的成果`, or an unqualified `机会全景`.

## 2. Sentence architecture

### 2.1 Lead with the business result

Prefer:

> `[客户]可用[交付物]在[决策点]减少[成本/时间/风险]`

over:

> `[算法/模型]基于[方法]实现[技术指标]`

Only show the technical mechanism after the business result, and only when it explains feasibility, differentiation, cost, or risk.

### 2.2 Use three-part reasoning

For a technical or market claim, write:

1. **已经知道什么**
2. **还不知道什么**
3. **因此先验证什么**

Example:

> 课堂实验已观察到师生脑信号同步；尚未证明这一指标能改善教学决策。因此，首个试点只验证学校是否会采用并付费，而不承诺提升成绩。

### 2.3 Use action-title grammar

Use one of these patterns:

- `[机会]兼顾[决定性条件A]和[条件B]，应作为首个切入口`
- `[关键证据]已支持[有限结论]，但尚不足以支持[更大投入]`
- `先以[模式A]取得付费证据，满足[里程碑]后再转向[模式B]`
- `[客户]愿否为[结果]付费，是下一阶段最重要的商业假设`
- `若[条件]未在[时间]内成立，应停止或更换方向`

Do not use a noun phrase as the whole title. Replace `市场分析` with the finding the analysis produced.

### 2.4 Prefer active, concrete sentences

Use:

- `模型评测团队向算法供应商交付一份可复核的现场测试报告。`
- `采购负责人只有在报告影响验收或上线决策时才有明确预算。`

Avoid:

- `面向多主体场景的多模态协同评测能力构建。`
- `相关能力有望形成场景化落地抓手。`

### 2.5 Keep one sentence to one job

- Put one claim or causal link in each sentence.
- Avoid more than two nested conditions.
- Avoid a chain of four or more nouns.
- Break long enumerations into a comparison, sequence, or appendix.
- Use a semicolon only when the two clauses have a direct `事实—含义` relationship.

## 3. Visible terminology

Use the right-hand language in core slides. The left-hand term may remain in the structured source, appendix, or first-use parenthesis when necessary.

| Internal, technical, or academic term | Core-slide expression |
| --- | --- |
| Lead Wedge | 首选切入点 / 首个验证方向 |
| Secondary Opportunities | 备选方向 |
| Go / Watch / Avoid | 建议推进 / 保留观察 / 暂不投入 |
| Opportunity Score | 相对验证优先级 / 为什么应先验证；不要把它写成市场规模、成功概率或已证实需求 |
| Product Thesis | 一句话产品主张：卖给谁、卖什么、改变什么结果 |
| User / Buyer / Budget Owner | 谁使用、谁拍板、谁付钱 |
| Job-to-be-Done | 客户必须完成的关键任务 |
| MVP | 第一版可售产品 / 首个可交付版本 |
| Product Form | 第一笔钱具体买什么 |
| Why Buy / Why Not Build | 为什么采购，而不是自建或继续手工做 |
| Commercialization Vehicle | 最合适的落地方式 / 由谁签约、交付并承担责任 |
| Pilot and Pricing Hypothesis | 首个试点与报价假设 |
| Productization Gap | 从研究成果到稳定交付还缺什么 |
| Defensibility | 为什么别人不容易复制 |
| Adoption Burden | 客户上线需要付出的时间、成本和流程变化 |
| Technical Readiness | 现在能稳定交付到哪一步 |
| Hard Blocker / Killer Risk | 足以叫停项目的风险 |
| Evidence Basis | 我们为什么这样判断 |
| Confidence | 当前结论的把握度 |
| Technical Evidence | 研究和原型中已经证明什么 |
| Market Evidence | 市场中已经观察到什么 |
| Inference | 本报告据此推断什么 |
| Interview Needed | 需要通过访谈确认什么 |
| Research Direction | 团队长期积累的能力 / 可解决的客户问题 |
| Research Output | 可复用的代码、数据、方法、硬件或交付经验 |
| Technology Route | 产品如何工作并产生客户价值 |
| External Validity | 换到真实客户现场后是否仍有效 |
| Generalization | 换人、换设备或换场景后还能不能用 |
| Robustness | 数据变脏或条件变化时是否还能稳定工作 |
| Incremental Validity | 相比客户现有方法，多带来多少有用信息 |
| Effect Size | 影响有多大，是否足以改变客户决策 |
| Statistical Significance | 结果不像随机出现；不等于商业影响足够大 |
| IP | 谁拥有使用、许可和收费的权利（IP） |
| SOP | 可复制的交付流程（SOP） |
| TAM / SAM / SOM | 理论市场 / 可服务市场 / 近期可拿下市场 |
| Technical Validation | 产品在目标场景能否稳定工作 |
| Commercial Validation | 客户是否愿意采购并持续使用 |

If a Chinese term is already clear, omit the English acronym. If an acronym is necessary for a later conversation, introduce the Chinese meaning first and the acronym in parentheses.

### 3.1 Scholar whole-picture language

Use the opening to establish context with these reader-facing labels:

- `现任职务与机构`
- `主要研究领域`
- `持续研究时间`
- `研究主线与演进`
- `代表成果`
- `学术影响力`

Apply these wording rules:

- Verify the current academic title from an official institution page and show the verification date.
- Calculate research duration from the first and latest relevant works when career-start data are unavailable. Write `公开成果显示至少持续研究X年`, not an unqualified `研究X年`.
- Name the citation database and retrieval date next to citation count or h-index. Do not mix Google Scholar, OpenAlex, Scopus, Web of Science, Semantic Scholar, or publisher counts in one comparison.
- Explain h-index on first use only when it adds value: `h-index为X，表示至少X篇成果各被引用X次`.
- Compare peer influence only when field, career stage, database, document types, and retrieval date are reasonably aligned. Otherwise show absolute metrics, representative highly cited work, and a limitation instead of a rank.
- Do not write `行业领先`, `领域头部`, `高被引学者`, or a percentile without a defined peer set and source.
- Treat citations, publication count, awards, and academic title as context for credibility and influence. Never present them as evidence that customers have demand or will pay.
- If reliable data are unavailable, write `公开数据未能可靠核实` and continue; do not estimate.

Preferred pattern:

> `截至[日期]，[数据库]收录[成果数]项成果、[引用数]次引用；影响力主要集中在[主题/代表成果]。该指标反映学术传播，不代表商业需求。`

### 3.2 Explain unfamiliar metrics on first use

Do not assume a commercially trained reader knows bibliometrics, technical benchmarks, clinical endpoints, or field-specific rankings. On first visible use, state:

1. **What it is**: the database, measurement, or benchmark.
2. **What the number literally means**: translate the value into one plain sentence.
3. **What it does not prove**: quality, demand, causality, rank, or commercial value unless separately supported.

Use these plain-language patterns:

- `Scopus 是国际学术文献数据库；125 表示其作者档案归集了 125 项文献。该数量可能受作者档案自动聚合影响，不代表 125 项高质量或可商业化成果。`
- `h=32 表示至少 32 篇成果各被引用 32 次以上。它反映一组成果的同行传播，不代表市场需求，也不能脱离学科和研究年限用于排名。`
- `引用 2,950 次表示该数据库记录到其他文献对这些成果的累计引用；不同数据库、学科和检索日期不可直接混比。`
- `期刊影响因子衡量期刊的平均引用表现，不等于某篇论文、某位学者或某项技术的质量。`

Prefer translating the metric in the action title. If the technical name remains in a card or note, place its explanation on the same slide; do not send the reader to the appendix to understand the headline.

## 4. Technical translation method

Translate each necessary technical concept through three questions:

1. **它是什么？** One everyday-language sentence.
2. **客户为什么在意？** Connect to a decision, workflow, cost, time, risk, quality, or experience.
3. **它还没证明什么？** State the boundary between research and product promise.

Example:

> `脑际耦合`是两个人的脑信号是否同步变化。学校可能用它观察课堂互动，但现有研究尚未证明它能稳定改善教学决策。

Use analogy only when it reduces cognitive load and remains accurate. Do not replace evidence with analogy.

Technical terms belong in the core deck only when they change one of these decisions:

- whether the product can work;
- whether it can be differentiated;
- whether it can be deployed or regulated;
- whether it can be delivered at acceptable cost;
- whether a customer will trust the result.

Otherwise move the term to the appendix or speaker notes.

## 5. Evidence and uncertainty language

Use clear evidence verbs:

- `论文中的受控实验显示……`
- `公开采购或客户案例表明……`
- `本报告据此推断……`
- `尚无公开证据证明……`
- `需要访谈确认：……`
- `若试点不能达到……，应停止……`

Avoid academic hedging that hides the decision:

- `在一定程度上`
- `或可`
- `一定意义上`
- `不排除可能`
- `具有潜在的应用价值`

Replace it with a bounded statement:

- `当前证据只支持受控试验，不支持客户现场承诺。`
- `该方向值得访谈验证，但不应进入产品开发。`
- `价格为工作假设，不是已验证的付费意愿。`

Do not use `显著` without saying whether it means statistically non-random, commercially material, or both. Never convert statistical significance into customer value without effect size, comparator, and workflow meaning.

## 6. Empty-language controls

Do not use the following words unless the same sentence names a specific object, mechanism, metric, or decision:

- 赋能
- 领先
- 重大突破
- 深厚积累
- 广阔前景
- 生态
- 闭环
- 抓手
- 范式
- 平台化
- 全栈
- 协同
- 显著提升
- 可广泛应用
- 具有潜力
- 商业价值巨大
- 行业痛点明显

Preferred replacements:

- `赋能客户` → `把报告周期从10天缩短到3天`
- `形成闭环` → `评测结果直接进入模型验收决策`
- `构建生态` → `与两类设备商和一家渠道伙伴共同销售`
- `行业领先` → `在同一公开数据集上比现有方案高出X个百分点`
- `平台化` → `把三项重复交付步骤做成客户可自助使用的产品`
- `具有潜力` → `满足哪些条件后值得继续投入`

Never use a buzzword as a substitute for customer, workflow, buyer, deliverable, or measurable result.

### 6.1 Never-visible internal copy

Remove these from user-facing artifacts:

- reader-persona labels such as `面向 MBA 毕业生`, `普通商业决策者`, or `咨询顾问视角`;
- prompt and workflow language such as `按 Skill 2.0`, `本页需回答`, `第 5 页前给出结论`, or `story spine`;
- internal schema keys such as `Lead Wedge`, `Go/Watch/Avoid`, and opportunity aliases; traceability IDs such as `P/S/C` may remain;
- production and QA notes such as `TODO`, `QA PASS`, font checks, overflow checks, model names, tool names, and script names.

Reader-facing uncertainty such as `需要验证：<具体事实>`, `买方假设`, and `当前把握度` is decision content and must remain.

## 7. Copy budget

Use these as editorial budgets, not reasons to truncate:

- Action title: one complete sentence, normally no more than two displayed lines.
- Major messages: normally two to four per slide; no more than three on a typical evidence slide.
- Body paragraph: no more than two displayed lines; convert longer logic into a chart, sequence, or separate slide.
- New technical terms: normally zero; at most one per core slide and only when decision-relevant.
- Unexplained acronyms: zero.
- Equal-weight categories: no more than four; otherwise create a hierarchy or appendix.
- Main visuals: one per slide.

Do not enforce the budget with `…`, `...`, smaller text, hidden overflow, or substring slicing. Rewrite or split.

## 8. Rewrite examples

| Unfriendly or academic copy | MBA-readable consulting copy |
| --- | --- |
| `科研方向概览` | `现有成果可归为三类，但只有一类接近客户付费条件` |
| `CLISA/DAEST可实现跨被试表征对齐` | `模型能识别多人脑电中的共同反应，但跨客户准确率仍待验证` |
| `外部效度与算法泛化仍待验证` | `模型在论文数据上有效，换到客户现场后是否稳定仍未确认` |
| `增量价值：哪个指标已证明能改变客户决策？` | `客户是否愿为“比现有方法更早发现风险”付费，是首个试点必须验证的假设` |
| `产品成熟度` | `现有代码可支持联合试验，但还不能直接交给客户使用` |
| `商业化机会评分矩阵` | `人因模型评测兼顾需求与交付可行性，应作为首个切入口` |
| `商业模式分析` | `首单应按项目收费；出现重复需求后再考虑订阅` |
| `技术壁垒` | `客户现场数据和校准经验是竞争者短期内最难复制的部分` |
| `市场空间广阔` | `近期可触达的是30家模型供应商，每家采购单位是一项验收项目` |
| `IP/合同问题` | `可交易的产权边界尚不清楚，签首单前需确认软件、数据和合作成果归属` |
| `风险与挑战` | `最大风险不是算法，而是无法取得可合法使用的客户现场数据` |
| `下一步工作建议` | `未来90天只做三件事：锁定客户、跑通试点、设定产品化门槛` |
| `该成果可应用于教育、医疗、消费等多个场景` | `先聚焦一个高频工作流程；跨行业扩张应等到首个场景付费后` |

## 9. Editorial QA

Fail visible copy when any answer is `No`:

1. Can a reader identify the customer, problem, offer, and action without reading the appendix?
2. Does the title state a conclusion rather than a topic?
3. Does the body prove, quantify, or qualify the title instead of repeating it?
4. Does every technical term change a commercial decision?
5. Is every necessary term explained in everyday Chinese on first use?
6. Does each important number include unit, period, comparator, source, and meaning?
7. Is each uncertainty written as a specific fact to verify rather than vague caution?
8. Does each recommendation name what to do now, what not to do, and when to change course?
9. Does each risk include a test and stop/change condition?
10. Can each slide be read once and paraphrased by an MBA-level generalist?
11. Has empty language been replaced by a named customer result or decision?
12. Is all visible copy complete, with no ellipsis, clipping, orphan, or broken term?
13. Can the deck be forwarded unchanged to the named scholar without exposing an internal audience, prompt, workflow, QA process, model, or tool?
14. Can a non-specialist explain every unfamiliar metric after one reading, including what it measures and what it cannot establish?
